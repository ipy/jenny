# Thinking Effort and Persistence

## Overview

Currently, `jenny` supports streaming and rendering `thinking` (Reasoning/Chain of Thought) blocks generated by models (like Claude 3.7 Extended Thinking or OpenAI o1/o3 reasoning). However, these thinking blocks are **not persisted** to the session transcript (`.jsonl`).

Failing to persist thinking blocks and omitting them when sending the conversation history back to the API causes critical issues:
1.  **Prompt Cache Misses:** Both Anthropic and OpenAI rely on exact prefix matching. Omitting previous thinking blocks breaks the prefix, reducing cache hit rates to near zero for multi-turn conversations and drastically increasing costs.
2.  **API Errors (P0):** Anthropic requires that if an assistant response contains a `tool_use` block, the exact unmodified `thinking` block from that turn MUST be included in the history alongside the `tool_result`. Failing to do so triggers a `400 Bad Request`.
3.  **Loss of Reasoning Continuity:** The model loses its prior context and deductions in subsequent turns.

This specification addresses the P0 requirement of persisting and transmitting thinking blocks, and outlines the P1 requirement for controlling thinking effort.

## P0: Thinking Persistence

### Transcript Modification

The `TranscriptEntry` struct in `internal/session/manager.go` must be updated to store thinking blocks and their associated metadata (like Anthropic's signature).

```go
type TranscriptEntry struct {
	Type      string    `json:"type"`
	Timestamp time.Time `json:"timestamp"`
	// ... existing fields ...
	Content   string    `json:"content,omitempty"`
	
	// New fields for Thinking
	Thinking  string    `json:"thinking,omitempty"`
	Signature string    `json:"signature,omitempty"`
}
```

### Persistence Logic (`internal/agent/engine_loop.go`)

When the API returns an assistant message containing a `thinking` block, the engine must extract the thinking text and signature (if present) and save them to the `session.TranscriptEntry`.

-   If multiple thinking blocks exist in a single turn (rare but possible), they should be concatenated or the internal structure must be updated to support an array of thinking blocks. Currently, Anthropic emits one thinking block per turn.

### API Request Construction (`internal/api`)

When reconstructing the API request (`messages` array) from the loaded transcript:

#### Anthropic (`provider_anthropic.go`)
If an `AnthropicMessage` of role `assistant` has a non-empty `Thinking` field, a `type: "thinking"` content block MUST be inserted **before** the `type: "text"` or `type: "tool_use"` blocks.

```json
{
  "role": "assistant",
  "content": [
    {
      "type": "thinking",
      "thinking": "...",
      "signature": "..."
    },
    {
      "type": "text",
      "text": "..."
    }
  ]
}
```
*Requirement:* The thinking block and its signature must be byte-for-byte identical to what was received to guarantee a cache hit and prevent 400 errors.

#### OpenAI (`provider_openai.go`)
If an `OpenAIMessage` of role `assistant` has a non-empty `Thinking` field, it must be mapped to the `reasoning_content` field.

```json
{
  "role": "assistant",
  "reasoning_content": "...",
  "content": "..."
}
```

## P1: Thinking Effort Control

Different API providers allow developers to constrain or dictate the amount of effort the model spends on reasoning. 

### Provider Abstraction

The internal `api.Message` or a new API request configuration struct (e.g. `ThinkingConfig`) should abstract these differences:

-   **Anthropic:** Uses `thinking.budget_tokens` (integer).
-   **OpenAI (o1/o3):** Uses `reasoning_effort` (enum: `"low"`, `"medium"`, `"high"`).

### CLI Interface (`internal/cli/cli.go`)

Add flags to expose this configuration to the user.

-   `--thinking <tokens>`: Sets the budget tokens for Anthropic. (e.g. `--thinking 4000`)
-   `--effort <level>`: Sets the reasoning effort for OpenAI. (e.g. `--effort high`)

*Open Question:* Whether to unify these into a single abstract flag or keep them separate. Keeping them separate is preferred initially to match provider semantics exactly.

### Provider Implementation

#### Anthropic
If `--thinking` is provided, the API client must include the `thinking` object in the top-level request:
```json
{
  "model": "claude-3-7-sonnet-20250219",
  "max_tokens": 20000,
  "thinking": {
    "type": "enabled",
    "budget_tokens": 4000
  },
  "messages": [...]
}
```
*Note:* Anthropic requires `max_tokens` to be strictly greater than `budget_tokens`. The API client must validate or automatically adjust `max_tokens` if this constraint is violated.

#### OpenAI
If `--effort` is provided, the API client must map it to the `reasoning_effort` field in the request:
```json
{
  "model": "o3-mini",
  "reasoning_effort": "high",
  "messages": [...]
}
```

## Acceptance Criteria

-   **AC1 (Persistence):** After a turn where the model generates a thinking block, inspecting the `.jsonl` transcript file reveals the `thinking` (and `signature` if applicable) fields populated in the `assistant` entry.
-   **AC2 (Anthropic Tool Use):** In a multi-turn conversation using Claude 3.7, if the model uses a tool and generates a thinking block, the subsequent API request containing the `tool_result` succeeds (HTTP 200) and does not return a 400 error regarding missing thinking blocks.
-   **AC3 (Prompt Caching):** For both OpenAI and Anthropic, subsequent turns in a conversation where thinking blocks are generated show a high number of `cache_read_input_tokens`, indicating that the prefix containing the thinking block matched successfully.
-   **AC4 (Effort Flags):** The `--thinking` and `--effort` CLI flags correctly populate the respective `budget_tokens` and `reasoning_effort` fields in the outgoing API JSON requests.
-   **AC5 (Backward Compatibility):** Loading older transcripts that lack the `thinking` and `signature` fields does not cause parsing errors or crashes.
